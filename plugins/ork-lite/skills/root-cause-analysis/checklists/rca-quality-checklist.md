# RCA Quality Checklist

Validate your Root Cause Analysis before closing.

## Problem Definition

- [ ] Problem statement is specific and measurable
- [ ] Timeline of events documented
- [ ] Impact quantified (users, revenue, time)
- [ ] Scope clearly bounded

## Investigation Quality

- [ ] Evidence-based (not assumption-based)
- [ ] Multiple hypotheses considered
- [ ] Hypotheses tested and documented
- [ ] Logs and data reviewed
- [ ] Code/config changes examined
- [ ] Timeline correlated with events

## Root Cause Identification

- [ ] Root cause is actionable (can be fixed)
- [ ] Root cause is within our control
- [ ] Fixing root cause would have prevented issue
- [ ] Root cause is not just a symptom
- [ ] Multiple contributing factors identified

## Method Selection

- [ ] Appropriate method used (5 Whys / Fishbone / Fault Tree)
- [ ] Method matches problem complexity
- [ ] Cross-functional input gathered (if needed)
- [ ] Sufficient time spent on analysis

## Blameless Culture

- [ ] Focus on systems, not individuals
- [ ] No names in blame context
- [ ] Psychological safety maintained
- [ ] Learning orientation emphasized

## Actions Defined

- [ ] Immediate fix implemented
- [ ] Preventive actions identified
- [ ] Each action has owner
- [ ] Each action has due date
- [ ] Actions are SMART (Specific, Measurable, etc.)

## Documentation

- [ ] RCA document completed
- [ ] Timeline included
- [ ] Evidence attached/linked
- [ ] Actions tracked in issue system
- [ ] Shared with relevant stakeholders

## Follow-Up

- [ ] Immediate fix verified
- [ ] Monitoring in place for recurrence
- [ ] Preventive actions tracked
- [ ] Retrospective scheduled (if major incident)
- [ ] Knowledge base updated

## Anti-Patterns Avoided

- [ ] NOT stopped at first cause found
- [ ] NOT blamed individuals
- [ ] NOT based on guesses
- [ ] NOT rushing to action without understanding
- [ ] NOT closed without preventive measures
