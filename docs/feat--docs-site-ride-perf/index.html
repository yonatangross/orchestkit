The factory ride: why it read as a static image

The scroll-world hero looked frozen and scrolled badly. Four causes, each measured rather than guessed. Step through them below.

101 frames 1280×716 master 360vh track zero animation deps

1. Every frame was requested at once

Crossing 1% of the track fired all 101 images simultaneously. They arrived in request order, not in the order the scrub needed, so the canvas sat on a stale frame while bytes for the far end of the flight were still in flight. Each cell is one frame.

Before0 in flight

2. Bounded, and aimed at the playhead

Now at most 6 requests are in flight, and the next one dispatched is always the queued frame nearest the current position. A fast scroll re-targets the network instead of waiting out a 101-deep queue.

After0 in flight

3. Scroll offset drove the frame index directly

With no easing, one trackpad tick jumped several frames at once. Rendered progress now eases toward the scroll target, so playback smoothness is decoupled from input granularity. Both heads below receive the identical stepped input.

Before · rawdiscrete jumps
After · easedcontinuous

4. Portrait had no overscan to pan within

Cover-fitting a 1280×716 frame into a 390×844 phone matches the height exactly, so the overflow is horizontal and there is no vertical slack. The source's empty upper third landed on screen as dead space. The fix scales past cover to manufacture overscan, then anchors into it. Geometry diagram, not a screenshot.
