MaceTenth/token-efficient-scripts — is it worth installing?

Every score below is anchored to a command that was actually run, not a reading of the README. The benchmark was reproduced twice from source; the guardrails were tested against real files. Verdict scores the plugin. Adoption shows what we take from it, and the two real ork bugs it led me to.

5.2
GRADE D
Don't install · harvest instead

The ideas are sound and the correctness guardrails are genuinely real (I verified all three against real files). But the apparatus around them doesn't hold up: the benchmark's equivalence oracle is provably unsound, the shipped harness re-verifies 4 of the 36 documented experiments, and the Stop hook spends ~0.5s of every session re-deriving a hardcoded constant. Worth reading once. Not worth installing — especially since you already run RTK.

What actually survives — the harvest list

✅The three correctness guardrails. Verified real: wc -l returned 2 on a 3-line file with no trailing newline; cat f1 f2 produced literal errorerror and undercounted 2→1; uniq alone said 3 uniques vs 2. These are worth stealing verbatim.
✅Its own epistemic honesty. It self-corrected compact-JSON down from 50–79% to the measured 40.64%, and states plainly that a universal monthly cost claim is "not established" — then the README prints ~94% anyway.
⚠️The self-improvement loop dead-ends. CLAUDE_PLUGIN_DATA is real (CC 2.1.78) and the log persists correctly — I checked, and my "it's vapor" hypothesis was wrong. But folding a finding back into SKILL.md is, by the plugin's own admission, "a maintainer step". The user's loop cannot close.
❌The oracle is unsound. num() takes the first integer in stdout: on access2.log:7 it reads 2 — the digit from the filename. On multi-file a.log:3\nb.log:99 it reads 3 and silently drops every later file. A same_answer=yes can come from reading the wrong number.
❌The test can't fail. Every experiment pits a deliberately terrible baseline (dump 20k rows) against the obvious right answer (grep -c). It authored both sides. A suite that cannot fail carries no information.
❌"Self-benchmarking weekly" isn't a thing. No CI, no .github/, no workflow anywhere. The one bench-bot commit says "(manual verification run)" in its own message and appended 9 lines of markdown.
❌MIT is claimed, not granted. plugin.json says "license":"MIT" and the README wears an MIT badge, but there is no LICENSE file and GitHub reports licenseInfo: null — so it's technically all-rights-reserved.
❌It leaks. bench.py:10 calls mkdtemp() with no rmtree — 2.73 MB orphaned per run; at the advertised weekly cadence that's ~142 MB/year.

1

Token estimator never reported a total at all

src/skills/audit-full/scripts/estimate-tokens.sh — found by auditing for the wc -l pattern, then got worse on contact

FIXED · WAS SEVERE
Before — today's code
count_lines() { lines=$(find … -print \ | xargs wc -l | tail -1 …) printf "row…" # display echo "$tokens" # return value } # SAME stdout! result=$(count_lines …) TOTAL=$((TOTAL + result)) # ← row text # → arithmetic syntax error # → TOTAL stays 0, exit 0
After — the fix
count_lines() { # value only find … -exec grep -ch '' {} + \ | awk '{s+=$1} END{print s+0}' } add_type() { # display only lines=$(count_lines "$ext") printf "row…" TOTAL=$((TOTAL + tokens)) } # one count per file, sums every # batch, counts lines not newlines

What I found on contact: I set out to fix a −94.7% undercount and found something worse. count_lines() wrote its table row and its return value to the same stdout, so the caller did arithmetic on the row, threw a syntax error, and left TOTAL=0 — while exiting 0. Every run answered "1M context: YES — tiny footprint" no matter the codebase. That's the one question the script exists to answer.

Before → after, run on this repo:
TOTAL on audit-full/  0 → 8,597 tokens  (+ two arithmetic errors gone)

2

500-line SKILL.md gate is off by one

src/skills/audit-skills/scripts/run-audit.sh:47 — the guardrail's literal case

FIXED · WAS MINOR
Before
LINE_COUNT=$(wc -l < "$SKILL_MD" \ | tr -d ' ') # wc -l counts NEWLINES. a file with # no trailing newline reports one # line fewer than it has
After
LINE_COUNT=$(grep -c '' "$SKILL_MD") # grep -c '' counts LINES — # correct with or without a # trailing newline

Why it matters: 7 of 114 SKILL.md files ship without a trailing newline. Each gets a silent 1-line discount against the 500-line limit. Only bites exactly at the boundary — a 501-line file reads as 500 and passes. Small, but it's a quality gate lying about the thing it gates.

Verified: two identical 500-line files — wc -l reports 499 for the newline-less one, 500 for the other.
Repo-wide: exactly 132 .md files lack a trailing newline — matching the 132-line delta between fix variants in bug 1 to the line.
3

cat across files glues boundary lines

No current ork hits — adopted as prevention

ADOPTED
Before — the trap
cat f1 f2 | grep -c error # f1 ends "error" with NO newline, # f2 starts "error" # → cat yields "errorerror" # → grep -c returns 1, truth is 2
After
grep -hc error f1 f2 \ | awk '{s+=$1} END{print s}' # per-file processing — boundaries # never merge → returns 2

Why we take it anyway: I grepped ork and found zero multi-file cat pipelines today. But the fix for bug 1 introduces exactly this shape (xargs -0 cat | wc -l was a candidate) — so the guardrail is what steered the fix to grep -ch instead. It earns its place by preventing the fix from creating the next bug.

Verified: cat f1 f2 produced the literal string errorerror; grep -c fell from the true 2 to 1.
4

sort before uniq

ork already does this correctly everywhere

DOCUMENTED

Every uniq in ork is already preceded by sort (checked explore/dependency-mapper.sh, emulate-seed/auto-discover.sh, and the rest). The blind refuter called this guardrail "near-universal knowledge bordering on filler" and our codebase agrees. We document it for completeness and change nothing.

5

Don't parse counts out of decorated stdout

The lesson from their num() bug — a test-practice rule, not theirs

ADOPTED
Their bug (don't copy)
num = re.findall(r"-?\d+", out)[0] # "access2.log:7" → 2 (!!) # grabs the digit from the FILENAME # "a.log:3\nb.log:99" → 3 # drops every later file
Our rule
grep -c … | tr -d ' ' # exact int(out.strip()) # typed # compare whole stdout, or parse a # known shape — never "first int # you can find in the output"

Why: this is the defect that sank their testability score to 2.5. We're not adopting their code — we're adopting the lesson, because ork's own audit scripts do awk '{print $1}' over decorated output in several places.

Verified: built the counterexample — num("access2.log:7") returns 2, not 7.

What we explicitly reject — and why

🚫The plugin itself. Scored 5.16/D/FAIL. Installing it adds a Stop hook, a per-invocation context cost, and a marketplace dependency to get advice its own "Tool selection order" says to skip in favour of native Grep/Glob.
🚫The Stop hook. ~0.5s of every session stop to re-derive a constant fixed by a hardcoded literal. Zero information after fire #1.
🚫The benchmark. No failure channel — forced all three assertions to DIFF and it still exited 0. Its fixtures all end with trailing newlines, so it cannot test the very guardrails we're adopting.
🚫Every headline number. "~94% saving" is marked "Not established" in their own evidence base. "1.68×" is contradicted by their own shipped log (1.52×) and by my runs (2.55×, 2.17×).
🚫Tier 3 (code trimming). Their own doc calls it "a rounding error", and they misapply it to bench.py — shipped, long-lived code — against their own stated scope.

The honest version of this adoption

The guardrail text did not name bug 1 — xargs batching is a different failure from "wc -l counts newlines". What actually happened: the guardrail worked as a search heuristic. Grepping ork for the pattern it warns about led me to a worse, unrelated bug sitting next to it.

That's a real result, and it's also the fair way to value this project: ~20 lines of prose that pointed at a −95% error in our own token estimator. Worth reading. Still not worth installing — and the estimator bug was ours, not theirs.
