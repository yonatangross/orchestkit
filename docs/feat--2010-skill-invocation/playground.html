🗃️ SQLite the OrchestKit way

Our M168 session registry runs on the built-in node:sqlite (Node ≥22.5), shared by every CC session at ~/.local/state/orchestkit/sessions.db. These three views are the actual correctness rules from session-registry.ts — and what #2006 adds on top. Toggle things and watch them pass or break. 🧪

open-time choices

The pragma order is load-bearing. node:sqlite has no default busy handler, unlike better-sqlite3 — so the WAL switch under concurrency throws instantly unless a timeout is already armed.

open sequence

write storm

Each session takes the RESERVED lock at BEGIN IMMEDIATE, so contention surfaces there — not mid-write. Losers see SQLITE_BUSY (errcode 5/261) and retry up to 15× with 20–150ms jitter (~1.3s worst case). Sync spin: hooks + DatabaseSync are synchronous.
