Quality Bar sections

effort S · impact XL

MengTo's flagship skill ends with falsifiable acceptance criteria. Ours end with link lists. Adding an explicit rubric to the top ~15 skills gives graders and /ork:verify something to score against — a direct attack on the weakest scorecard axis (Eval, 4/10).

Beforesrc/skills/cover/SKILL.md
## Related Skills - ork:verify — grade the generated tests - ork:implement — the step before this one (file ends — no acceptance criteria; "done" is whatever the model felt like)
Aftersrc/skills/cover/SKILL.md
## Quality Bar Done means all of these hold — not "tests were written": - Every public function touched by the diff has ≥1 test - Failure paths tested, not just happy paths - Suite passes locally: paste the summary line as proof - No test asserts on implementation details (mocks of internals) - Coverage delta reported vs base branch ## Related Skills - ork:verify — grade the generated tests

Video input mode for design-context-extract

effort M · impact L

Confirmed gap: the skill accepts screenshots and URLs only — zero video coverage. Motion (easing, scroll behavior, transitions) is exactly what a screenshot can't carry. MengTo's video-to-superprompt pipeline maps cleanly onto our ffmpeg-capable Bash + multimodal Read.

Beforedesign-context-extract
/ork:design-context-extract /tmp/shot.png /ork:design-context-extract https://example.com /ork:design-context-extract current project # Input types: screenshot | URL | project # A screen RECORDING? → unsupported. # Motion, easing, scroll choreography: lost.
Afterdesign-context-extract
/ork:design-context-extract /tmp/recording.mp4 # New: video pipeline ffprobe → duration, fps, dimensions ffmpeg → frames at timeline beats (not uniform) analyze → layout / motion / visual layers output → tokens + motion-spec.md (durations, easing, scroll triggers, reduced-motion fallbacks)

design-stylecards skill

effort M · impact L

"Specs beat vibes." MengTo encodes each aesthetic as literal copy-paste values with Use-When and Avoid blocks. Our design skills are procedural (how to run a pipeline) — none are declarative taste assets. One skill + a references/ catalog of named recipes gives design-to-code a reusable vocabulary.

Beforesrc/skills/
design-context-extract/ # pipeline design-to-code/ # pipeline design-system-tokens/ # architecture # Ask for "polished elevation" and the # model invents shadows from vibes — # different every session.
Afterdesign-stylecards/references/
### Layered neutral elevation — md Use when: cards, panels, popovers. shadow-[0px_0px_0px_1px_rgba(0,0,0,.06), 0px_3px_3px_-1.5px_rgba(0,0,0,.06), 0px_12px_12px_-6px_rgba(0,0,0,.06)] Avoid: mixing with default shadow scale; colored tints; stacking on one element.

CI gate: quality-bar presence

effort S · impact M

Scope pivot after checking the repo: a trigger-phrase description lint already exists (test-description-triggers.sh), and MengTo's links-only-references rule doesn't map — ork's references/ are progressive-disclosure docs by design. The genuinely missing gate: once the top skills carry a Quality Bar, nothing stops a refactor from silently dropping it.

Beforetests/skills/
test-description-triggers.sh # exists ✓ test-skill-length.sh # 520 cap ✓ test-rubric-schema.sh # rubric.json ✓ # Nothing asserts a skill keeps its # Quality Bar section after edits.
Aftertests/skills/structure/
test-quality-bar.sh # For each skill in the covered list: grep -q "^## Quality Bar" SKILL.md || FAIL "rubric dropped" # List grows as more skills adopt it.

Execution order

Playground→ WS1 · 15 skills (3 parallel agents) ∥ WS2 · video mode ∥ WS3 · stylecards → WS4 · CI gate→ build · test · PR

WS4 depends on WS1 (the gate asserts what WS1 adds). WS1–WS3 touch disjoint files and run in parallel.

Risks & pre-mortems

medium

Line-cap collision

implement is at 510/520 lines. Its Quality Bar must be ≤8 lines or trim elsewhere first. Mitigation: per-skill wc -l check after each edit.

low

Generic rubrics

A copy-pasted Quality Bar is worse than none. Mitigation: each rubric written against the skill's own output contract; reviewed in one diff.

low

Token ceiling

Skills near the 8000-token ceiling could tip over. Mitigation: test-skill-tokens.sh runs in the same suite — failure is loud, pre-push.

Decisions taken

Links-only references: rejected

ork's references/ are load-on-demand docs, not link lists. Imposing MengTo's rule would break progressive disclosure. Documented, not adopted.

Trigger lint: already exists

test-description-triggers.sh covers it. WS4 re-scoped to the quality-bar presence gate instead of duplicating.

One stylecards skill, not 20

MengTo ships ~20 mood micro-skills. We ship one skill with a references/ catalog — same taste density, no manifest sprawl.

Apply the MengTo/Skills adoption plan: add Quality Bar rubrics to the top 15 ork skills, a video input mode to design-context-extract, a design-stylecards skill with an exact-value recipe catalog, and a CI gate asserting Quality Bar presence.
