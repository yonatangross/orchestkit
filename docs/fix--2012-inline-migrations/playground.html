🚚 Why SQLite migrations didn't run in the shipped plugin

The M168 session registry reads its NNN-*.sql migrations from disk at runtime — from the bundle's own directory (dist/). But rsync excludes src/ where the .sql live, and esbuild doesn't inline them. So in the distributed plugin the runner found zero files and created no tables — silently. Tests stayed green because vitest runs the source tree. Toggle the build step and trace it. (#2012)

build pipeline

At runtime MIGRATIONS_DIR = dirname(import.meta.url) = dist/, and run-hook.mjs imports bundles from dist/. So the .sql must exist in dist/ — by being inlined, shipped via src/, or copied in.
