Telemetry event field: '' instead of the real hook name

Two independent nullish-coalescing gaps compounded into a silent hole: 91% of local telemetry rows carried event:"" instead of a real hook name — invisible to any sink with a non-empty allowlist. Root-caused via CC's official docs (hook_event_name is guaranteed on every invocation), not guessed.

Verification trail

Root cause confirmed, not assumed — consulted the official CC hooks docs via a specialist agent: hook_event_name is a guaranteed field on every hook invocation, all event types. Ruled out "CC doesn't always send it" before writing any fix.
Live-capture attempt made and abandoned — instrumented the actual installed hook dispatcher for a raw-payload capture (same technique as prior SubagentStart investigations), hit repeated environment issues (sandboxed /tmp, async/await syntax risk to a live session), and reverted cleanly (byte-identical MD5 to the pre-edit backup) once the safer documentation-based path confirmed the answer.
Existing test gap identified — prior tests covered the undefined case only (which ?? correctly catches), never the empty-string case — exactly the gap that let this ship unnoticed. Both new tests assert !== '' explicitly, not just === 'unknown'.
