Next.js route export rules — the buildLlmsTxt fix

PR #2337. A Next.js route.ts module may only export request handlers + reserved config. #2329 exported a shared helper from a route file → the Docs Site Build's route-type check failed and reddened main. Fix: move the helper to lib/.

✅ What a route module is allowed to export

GETPOSTPUT PATCHDELETEHEAD OPTIONSdynamicrevalidate runtimefetchCachepreferredRegion maxDurationgenerateStaticParams

Anything else — an arbitrary helper like buildLlmsTxt() — fails: "Route … does not match the required types of a Next.js Route."

BEFORE (#2329) reddened main

// app/api/llms.txt/route.ts
export const revalidate = false;

// ❌ non-handler export from a route module
export function buildLlmsTxt(): string { … }

export function GET() {
  return new Response(buildLlmsTxt(), …);
}

// app/api/llms.txt.md/route.ts
import { buildLlmsTxt } from
  "@/app/api/llms.txt/route"; // ❌ imports a route

AFTER (#2337) build green

// lib/llms-txt.ts  ✅ plain module
export function buildLlmsTxt(): string { … }

// app/api/llms.txt/route.ts
import { buildLlmsTxt } from "@/lib/llms-txt";
export const revalidate = false;
export function GET() { … }

// app/api/llms.txt.md/route.ts
import { buildLlmsTxt } from "@/lib/llms-txt"; ✅

🔎 Why it slipped past auto-merge

Docs Site Build is not a required check, so #2329/#2331/#2333/#2334 merged with it red. And a local next build --webpack on a warm .next cache skips the route-type validation — a clean build (fresh worktree) reproduces and confirms the fix.
